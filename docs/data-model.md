# Data Model

This document describes the structure of the report JSON generated by `fetch_from_api.py` or `parse_log.py`. The same structure is consumed by `index.html` for rendering.

## Top-Level Structure

```json
{
  "players": {
    "PlayerName": { "role": "Tank|Healer|DPS", "class": "Warrior|Paladin|..." }
  },
  "bosses": {
    "Boss Name": {
      "total_pulls": 5,
      "kills": 1,
      "wipes": 4,
      "pulls": [ <Pull>, <Pull>, ... ]
    }
  }
}
```

## Field Notes

### deaths

Sorted by time of death. The first entry is the first player to die on the pull.

### player_damage_taken

Keys are formatted as `"Source: Ability"` with hit counts as values (not damage amounts). Used for mechanic failure detection.

### heal_details

Per-healer, per-spell breakdown. `total` is effective healing, `overheal` is wasted healing. `is_hot` indicates if the spell is a heal-over-time.

### buff_events

Chronological list of buff applications, removals, and refreshes. Types: `applybuff`, `removebuff`, `refreshbuff`, `applydebuff`, `removedebuff`.

### clutch_heals

Only present if fetched with the updated `fetch_from_api.py`. Sorted by HP% ascending (most clutch first). `hp_pct` is the target's HP percentage *before* the heal landed.

### biggest_heals / biggest_crits

Top 5 per pull, sorted by amount descending. Only present with the updated fetcher.

### consumables

Currently empty (`{}`) when fetched via the API. Only populated from raw combat log parsing via `parse_log.py`.

## Data Source Differences

| Feature | API (`fetch_from_api.py`) | Raw Log (`parse_log.py`) |
|---------|--------------------------|--------------------------|
| Per-event healing (clutch heals, biggest hits) | Yes | Yes |
| Buff events | Yes | Yes |
| Consumable tracking | Limited (empty) | Full |
| Talent spec detection | No | Yes |
| Player positioning | No | Partial |
| Damage amounts per hit | Only via `biggest_crits` | Yes |
| Phase detection | Via creature deaths | Via combat log events |
